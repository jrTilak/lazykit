## Props

- `createStrongPasswordSchema(options)` — Creates the schema. The exported
  `strongPasswordSchema` uses the default empty options.
- `options.blocklist` (`ReadonlySet<string>`, optional) — Complete password
  values to reject. Entries are NFC-normalized once when the schema is created.
- `input` (`string`) — The password to normalize and validate.

Unknown options, non-set blocklists, and blocklists containing non-string or
malformed Unicode values throw when the schema is created.

### Returns

`StrongPassword`, an NFC-normalized string branded with
`"ValidatedPassword"`, `"Password"`, and `"StrongPassword"`. It contains 15
to 128 Unicode code points, so it remains assignable to the default `Password`
type. Valid surrogate pairs count as one code point; unpaired UTF-16 surrogates
are rejected. Leading and trailing whitespace is preserved.

Blocklist matching compares the complete normalized password exactly and is
case-sensitive. It does not reject substrings or variants that are not present
in the set. Supply a current breached/common-password list and relevant
service-specific values when creating or changing a password. At sign-in,
perform the same NFC normalization before hashing or comparison without
reapplying a newer blocklist to an existing credential.

The preset intentionally has no uppercase, lowercase, number, or symbol
requirements. This follows [NIST SP 800-63B-4](https://pages.nist.gov/800-63-4/sp800-63b/authenticators/),
which emphasizes length and blocklist checks and disallows password composition
rules. Passing this schema alone is not proof that a password is resistant to
guessing.
